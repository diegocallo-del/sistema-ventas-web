¿QUÉ ES GIT?
═══════════════════════════════════════════════════════════════════
Git es un sistema de control de versiones que permite:
✓ Guardar historial de cambios del código
✓ Trabajar en equipo sin pisarse el código
✓ Volver a versiones anteriores si algo sale mal
✓ Ver quién hizo cada cambio y cuándo
✓ Trabajar en ramas paralelas

FLUJO DE TRABAJO DIARIO
═══════════════════════════════════════════════════════════════════
CADA VEZ QUE VAYAS A TRABAJAR, SIGUE ESTOS PASOS:
PASO A: ACTUALIZAR TU CÓDIGO (PULL)
────────────────────────────────────
ANTES de empezar a trabajar, descarga los cambios de otros:

Abrir terminal en la carpeta del proyecto
Ejecutar:
git pull origin main

Esto descarga todos los cambios que otros subieron.
PASO B: TRABAJAR EN TU CÓDIGO
──────────────────────────────
Realiza tus cambios normalmente:

Crear clases
Modificar archivos
Agregar código

PASO C: VER QUÉ ARCHIVOS CAMBIASTE
───────────────────────────────────
Antes de subir, revisa qué modificaste:
git status
Te mostrará:

Archivos modificados (rojo)
Archivos nuevos (rojo)
Archivos listos para commit (verde)

PASO D: AGREGAR TUS CAMBIOS (ADD)
──────────────────────────────────
Agregar archivos específicos:
git add ruta/archivo.java
O agregar TODO:
git add .
Verificar:
git status
(ahora deben estar en verde)
PASO E: HACER COMMIT (GUARDAR CAMBIOS)
───────────────────────────────────────
Guardar tus cambios con un mensaje descriptivo:
git commit -m "Descripción clara de lo que hiciste"
Ejemplos de buenos mensajes:
git commit -m "Agregado ProductoDAO con métodos CRUD"
git commit -m "Corregido error en cálculo de IGV"
git commit -m "Implementado servicio de ventas"
git commit -m "Agregada validación de DNI"
Ejemplos de MALOS mensajes (NO HACER):
git commit -m "cambios"
git commit -m "fix"
git commit -m "asdf"
PASO F: SUBIR TUS CAMBIOS (PUSH)
─────────────────────────────────
Subir tus commits al servidor:
git push origin main
Si todo sale bien, tus cambios ya están en GitHub.

TRABAJAR CON RAMAS (OPCIONAL)
═══════════════════════════════════════════════════════════════════
Las ramas permiten trabajar en paralelo sin afectar el código principal.
CREAR UNA RAMA:
git branch nombre-rama
git checkout nombre-rama
O en un solo comando:
git checkout -b nombre-rama
Ejemplo:
git checkout -b feature-productos
TRABAJAR EN LA RAMA:

Hacer cambios
git add .
git commit -m "mensaje"
git push origin nombre-rama

VOLVER A LA RAMA PRINCIPAL:
git checkout main
FUSIONAR RAMA CON MAIN:
git checkout main
git merge nombre-rama
SUGERENCIA DE RAMAS POR PERSONA:

PERSONA 3: rama "daos"

RESOLVER CONFLICTOS
═══════════════════════════════════════════════════════════════════
Un conflicto ocurre cuando dos personas modifican las mismas líneas
de un archivo.
CÓMO SE VE UN CONFLICTO:
Al hacer git pull, verás un mensaje de conflicto.
CÓMO RESOLVERLO:

Abrir el archivo con conflicto

Verás algo así:
<<<<<<< HEAD
tu código
código de otro







rama








Decidir qué código mantener:

Mantener solo el tuyo
Mantener solo el del otro
Combinar ambos


Eliminar las marcas <<<<<<, =======, >>>>>>>
Guardar el archivo
Agregar y hacer commit:
git add .
git commit -m "Resuelto conflicto en archivo.java"
git push origin main

EVITAR CONFLICTOS:
✓ Hacer git pull ANTES de empezar a trabajar
✓ Hacer commits pequeños y frecuentes
✓ Comunicar en qué archivos estás trabajando
✓ No modificar archivos de otros sin avisar
═══════════════════════════════════════════════════════════════════
REGLAS DEL EQUIPO PARA GIT
═══════════════════════════════════════════════════════════════════

SIEMPRE hacer git pull ANTES de empezar a trabajar
NO subir archivos compilados (.class, carpeta build/)

El .gitignore ya los excluye


NO subir archivos de configuración local

database.properties con tu password


Hacer commits con mensajes DESCRIPTIVOS
Hacer commits PEQUEÑOS y frecuentes

Mejor 5 commits pequeños que 1 grande


NO hacer push si el código no compila
Probar tu código ANTES de hacer push
Avisar al equipo cuando subas cambios importantes
Hacer git pull varias veces al día
Si tienes dudas, PREGUNTAR antes de hacer push

COMANDOS DE EMERGENCIA
═══════════════════════════════════════════════════════════════════
DESHACER CAMBIOS NO GUARDADOS:
git checkout -- archivo.java
(descarta cambios en ese archivo)
DESHACER EL ÚLTIMO COMMIT (sin perder cambios):
git reset --soft HEAD~1
DESHACER EL ÚLTIMO COMMIT (perdiendo cambios):
git reset --hard HEAD~1
⚠️ CUIDADO: No se puede recuperar
VER QUIÉN MODIFICÓ CADA LÍNEA:
git blame archivo.java
VOLVER A UN COMMIT ANTERIOR:
git log  (copiar el hash del commit)
git checkout hash-del-commit
